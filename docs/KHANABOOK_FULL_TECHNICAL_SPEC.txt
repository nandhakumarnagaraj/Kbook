────────────────────────────────────────────────────────────────────────────────
3.1 AUTHENTICATION WORKFLOW
────────────────────────────────────────────────────────────────────────────────

SIGNUP FLOW:
  User → Phone + Password + OTP (WhatsApp delivery)
  1. Android sends signup request to /auth/signup
  2. Server validates OTP via PasswordResetOtpService
  3. Server creates User + RestaurantProfile (sets lastResetDate)
  4. Server returns JWT token (30-day expiry)
  5. Android stores token in EncryptedSharedPreferences
  6. Triggers InitialSync → pulls server data to Room

GOOGLE SIGN-IN FLOW:
  1. Android uses Credential Manager → gets Google ID token
  2. Sends token to /auth/google
  3. Server verifies with Google API Client (google-api-client 2.2.0)
  4. Creates/updates User, returns JWT
  5. Android proceeds to InitialSync

LOGIN FLOW:
  1. Phone + Password → /auth/login
  2. Server validates credentials (bcrypt hashed)
  3. Returns JWT + user role
  4. Android: atomic database switch (different restaurant = fresh DB)
  5. SessionManager stores session, triggers sync

APP LOCK:
  Manager: SessionManager.kt
  - PIN stored: EncryptedSharedPreferences (AES-256)
  - Biometric: BiometricPrompt + KeyStore
  - Grace period: 30 seconds (GPay-style)
  - Trigger: app goes to background → lock on resume
  - Fallback: PIN always available if biometric fails

TOKEN LIFECYCLE:
  - 30-day expiry (server-side JwtService)
  - Token blocklist (TokenBlocklist entity + cleanup scheduler)
  - 401 response → redirect to LoginScreen
  - No refresh token (re-login required on expiry)

────────────────────────────────────────────────────────────────────────────────
3.2 BILLING WORKFLOW (Core Business Logic)
────────────────────────────────────────────────────────────────────────────────

BILL CREATION (NewBillScreen → Multi-step):
  Step 1: Menu Selection
    - LazyVerticalGrid shows menu items (responsive columns)
    - Tap item → add to cart (BillItemEntity)
    - Variant selection dialog if item has variants
    - Search/filter by category
    - Split-view on tablets (menu left, cart right)

  Step 2: Cart Review
    - Quantity +/- adjustment
    - Special instructions per item (specialInstruction field)
    - Cart total with GST calculation (BillCalculator.kt)

  Step 3: Payment
    - PaymentModeManager determines available modes
    - Modes: Cash, Manual UPI (QR), POS Terminal
    - Cash: enter received amount → calculate change
    - UPI: display QR (ZXing generation), mark paid manually
    - Split payment: divide across modes

  Step 4: Settlement
    - BillEntity status: DRAFT → COMPLETED
    - OrderIdManager generates: daily order number + lifetime order
    - TerminalDailyCounterEntity: atomic per-terminal invoice numbering
    - Invoice series format: T{terminalId}-{dailyCount}/{date}
    - Triggers immediate sync push to server
    - Print receipt (BluetoothPrinterManager)
    - Print KOT to kitchen (KitchenPrintQueueManager)

DATA MODEL (Room entities):
  BillEntity:
    - id, localId, serverId, restaurantId, terminalId
    - orderNumber, lifetimeOrderId, invoiceNumber
    - status (DRAFT/COMPLETED/CANCELLED/PAID)
    - orderType (DINE_IN/TAKEAWAY)
    - totalAmount, taxAmount, discountAmount
    - paymentMode, paidAmount, changeAmount
    - createdAt, updatedAt, syncedAt
    - isDeleted, recordOrigin, ownerTerminalId

  BillItemEntity:
    - billId → BillEntity (FK)
    - menuItemId, variantId
    - quantity, unitPrice, totalPrice
    - specialInstruction, sentToKot (boolean)

  BillPaymentEntity:
    - billId → BillEntity (FK)
    - mode (CASH/UPI/POS), amount
    - transactionReference

GST CALCULATION (BillCalculator.kt):
  - Subtotal = sum(item.quantity * item.unitPrice)
  - Tax = subtotal * taxRate (configurable per restaurant)
  - CGST = tax / 2, SGST = tax / 2
  - Total = subtotal + tax - discount

3.3 SYNC ENGINE WORKFLOW (Offline-First)
────────────────────────────────────────────────────────────────────────────────

COMPONENTS:
  Android: MasterSyncWorker.kt (WorkManager) → MasterSyncProcessor.kt
  Server:  MasterSyncController.java → GenericSyncService (sync/)

SYNC TRIGGER POINTS:
  1. Periodic: WorkManager enqueues every 15 minutes
  2. Immediate: on bill creation, payment, or manual trigger
  3. App startup: after login/InitialSync
  4. Network recovery: connectivity change listener

PUSH FLOW (Local → Server):
  1. MasterSyncProcessor queries Room for unsynced records
     (syncedAt == null OR updatedAt > syncedAt)
  2. Groups by entity type: Bills, BillItems, Payments, Menu, Users, etc.
  3. POST /sync/push/{entity} with batch payload + X-Terminal-Token header
  4. Server validates tenant isolation (restaurantId matches JWT)
  5. Server deduplicates (checks existing serverId)
  6. Server returns: { successfulLocalIds: [], localToServerIdMap: {} }
  7. Android updates local records with serverId + syncedAt timestamp
  8. Conflict: 409 → LWW (Last-Write-Wins based on updatedAt)

PULL FLOW (Server → Local):
  1. GET /sync/pull/{entity}?cursor={lastSyncTimestamp}&limit=100
  2. Server returns records updated after cursor, scoped to restaurant
  3. Android inserts/updates Room records
  4. If hasMore=true, continue pulling (pagination)
  5. Update sync cursor for next pull
  6. Post-pull reconciliation: resolve FK relationships

CONFLICT RESOLUTION:
  - Strategy: Last-Write-Wins (LWW) by updatedAt timestamp
  - Terminal isolation: ownerTerminalId prevents cross-terminal edits
  - Invoice series: unique constraint → 409 → increment and retry
  - Quarantine: unresolvable conflicts → SyncQuarantineEntity

MULTI-TENANT ISOLATION:
  - Every query scoped by restaurantId (WHERE restaurantId = ?)
  - Atomic DB switch on login to different restaurant
  - Server: tenant extracted from JWT, enforced on all operations
  - Cross-tenant push blocked with 403

────────────────────────────────────────────────────────────────────────────────
3.4 MULTI-TERMINAL MANAGEMENT WORKFLOW
────────────────────────────────────────────────────────────────────────────────

DEVICE REGISTRATION:
  1. App installed on new device → generates unique deviceId
  2. POST /terminal/activate with deviceId + restaurantId
  3. Server checks: MAX_ACTIVE_TERMINALS = 5 (pessimistic lock on profile)
  4. If < 5: creates DeviceRegistrationRequest (status: PENDING)
  5. SHOP_ADMIN approves via web-admin or TerminalManagementController
  6. On approval: creates RestaurantTerminal, returns terminalId + token
  7. Android stores X-Terminal-Token for all subsequent requests

TERMINAL RECLAIM (reinstall):
  1. Same deviceId re-activates → /terminal/activate
  2. Server: findMostRecentByRestaurantIdAndDeviceId
  3. If APPROVED request exists → return existing terminalId+token directly
  4. No infinite PENDING loop (fixed: return APPROVED immediately)

BILL ISOLATION:
  - Bills tagged with ownerTerminalId on creation
  - Visibility: each terminal sees only its own bills in draft/active
  - Completed bills visible to all terminals (for reports)
  - Server enforces: X-Terminal-Token must match ownerTerminalId

INVOICE NUMBERING:
  - TerminalDailyCounterEntity: { terminalId, date, counter }
  - Atomic increment per terminal per day (no collisions)
  - Format: T{id}-{counter}/{DDMMYYYY}
  - Seeded from existing bills on migration/APK update

ROLES:
  - OWNER: full access, manages restaurant
  - SHOP_ADMIN: approves terminals, manages staff terminals
  - USER: standard POS operator on assigned terminal

────────────────────────────────────────────────────────────────────────────────
3.5 PRINTING WORKFLOW
────────────────────────────────────────────────────────────────────────────────

PRINTER ARCHITECTURE:
  BluetoothPrinterManager → PrintRouter → PrintService (background)
  Kitchen: KitchenPrintQueueManager → KitchenTicketFormatter

RECEIPT PRINTING:
  1. Bill settled → PrintRouter routes to receipt printer (by MAC address)
  2. PrintService formats ESC/POS commands (80mm width)
  3. Bluetooth SPP connection → send byte stream
  4. Includes: shop name, logo (cached bitmap), items, GST, QR code
  5. Retry on failure (background service, non-blocking)

KOT (Kitchen Order Ticket):
  1. New items added → check sentToKot flag per BillItem
  2. Only unsent items printed (delta printing)
  3. KitchenTicketFormatter: item name, qty, special instructions
  4. Routed to kitchen printer (separate MAC)
  5. KitchenPrintQueueEntity tracks pending prints
  6. ReprintKdsScreen: view/reprint pending KOTs

PDF INVOICE:
  Manager: InvoicePDFGenerator.kt (uses PdfBox-Android 2.0.27.0)
  - Generates styled PDF: header, logo, items table, GST split, footer
  - Stored temporarily for sharing via WhatsApp/email

QR CODE:
  Manager: QrCodeManager.kt (uses ZXing 4.3.0)
  - UPI QR: encodes upi://pay?pa={vpa}&am={amount}&pn={name}
  - Invoice QR: deep link to public invoice URL

3.6 REPORTS & EXPORT WORKFLOW
────────────────────────────────────────────────────────────────────────────────

REPORT GENERATION (ReportGenerator.kt):
  - Queries Room: BillEntity + BillItemEntity + BillPaymentEntity
  - Filters: date range, order type, payment mode
  - Aggregates: total revenue, order count, avg ticket, top items
  - Time periods: Today, This Week, This Month, Custom Range

EXPORT (ReportExporter.kt):
  PDF Export:
    - Uses PdfBox-Android
    - Styled table: headers, rows, totals
    - Logo cached (Coil → Bitmap → stored for reuse)
    - Shares via Android Intent (WhatsApp, email, etc.)

  Excel Export:
    - CSV format with headers
    - Columns: Order#, Date, Items, Amount, Payment Mode, Status

INVOICE SHARING:
  1. Bill completed → serverId assigned after sync
  2. Invoice URL: https://kbook.iadv.cloud/api/v1/invoice/{token}
  3. Share via WhatsApp: PDF attachment + deep link
  4. Server InvoiceController serves public HTML invoice page

────────────────────────────────────────────────────────────────────────────────
5.1 EASEBUZZ PAYMENT GATEWAY WORKFLOW
────────────────────────────────────────────────────────────────────────────────

SUB-MERCHANT ONBOARDING:
  1. Restaurant owner submits KYC docs (PAN, GSTIN, bank details)
  2. Android uploads docs to /business/kyc/upload (local CDN)
  3. Server submits to Easebuzz Sub-Merchant API
  4. Status: PENDING_KYC → ACTIVE (or FAILED → allow resubmit)
  5. Web-admin: Business Directory shows KYC status + timeline

PAYMENT FLOW:
  1. Customer selects "Pay Online" at checkout
  2. Android checks sub-merchant status (ACTIVE required)
  3. If not ACTIVE: fallback to parent merchant
  4. Launches Easebuzz SDK (in.easebuzz:android-v2:1.0.6)
  5. SDK opens WebView with payment options (UPI, cards, netbanking)
  6. User completes payment → Easebuzz callback
  7. Android receives result → marks bill as PAID
  8. Server webhook confirms (HMAC-authenticated)
  9. Transaction recorded for commission split

SPLIT PAYMENT & COMMISSION:
  1. Each transaction → post-transaction split calculation
  2. Async commission service: platform fee % deducted
  3. Net amount → sub-merchant payout queue
  4. Payout tracking per sub-merchant in web-admin
  5. Commission reports generated for platform admin

REFUND FLOW:
  1. Owner initiates refund (web-admin or Android)
  2. If Easebuzz payment → gateway refund (capped to paid amount)
  3. If manual payment → manual refund only
  4. Refund status: INITIATED → PROCESSED → COMPLETED
  5. Webhook updates status async

WIRE INTEGRATION:
  - Lookup sub-merchant details from Easebuzz WIRE platform
  - Auto-match by email or subMerchantId
  - Web-admin UI: inline lookup results per merchant

5.2 PUSH NOTIFICATIONS WORKFLOW (FCM)
────────────────────────────────────────────────────────────────────────────────

SETUP:
  Server: Firebase Admin SDK (9.3.0) + service account credentials
  Android: Firebase Messaging (24.1.1) + FCM service

REGISTRATION:
  1. App startup (if logged in) → force register FCM device token
  2. After login/signup → register token with retry
  3. Token stored on server per (userId + deviceId)
  4. Stale tokens filtered to prevent cross-tenant 403

NOTIFICATION EVENTS (server triggers):
  - LOGIN_WELCOME: new user welcome message
  - NEW_ORDER: marketplace order received
  - PAYMENT_CONFIRMED: payment success notification
  - ORDER_CANCELLED: cancellation alert
  - KYC_STATUS: KYC approval/rejection update
  - PAYOUT_COMPLETED: money transferred notification
  - FSSAI_EXPIRY: license renewal reminder (action buttons)

DELIVERY:
  1. Server event fires (e.g., payment confirmed)
  2. PushNotificationService builds payload (title, body, data, deepLink)
  3. Firebase Admin sends to device token
  4. Android FCMService receives in background/foreground
  5. Notification channel: Orders | Payments | System | Promotions
  6. Deep link opens relevant screen on tap

RELIABILITY:
  - BootReceiver: re-registers token on device restart
  - OEM auto-start config: guides user to enable background
  - OAuth2 refresh token fallback for Firebase (org policy workaround)
  - Channel groups registered before channel creation (prevents crash)

TEMPLATES:

  - Test endpoint: POST /notifications/test

SECTION 9: SECURITY ARCHITECTURE
================================================================================

v1 SECURITY LAYERS:
  1. Network: SSL/TLS (Let's Encrypt) + certificate pinning on Android
  2. Transport: HTTPS enforced for all API calls
  3. Auth: JWT Bearer token in Authorization header
  4. Terminal: X-Terminal-Token header for bill ownership
  5. Rate Limiting: Bucket4j (login: 5/min, OTP: 3/min, API: 100/min)
  6. Database: SQLCipher AES-256 encryption on device
  7. Secrets: EncryptedSharedPreferences (PIN, tokens)
  8. App Lock: PIN + Biometric with 30s grace period
  9. Swagger: disabled in production (KB-008)
  10. Audit: SecurityAuditEvent logged for sensitive operations

v2 ADDITIONS:
  11. Webhook Auth: HMAC-SHA256 verification for Easebuzz callbacks
  12. Cache: Caffeine with TTL to prevent stale data exposure
  13. Email: server-side templates (no user input in body)
  14. Firebase: service account credentials (env vars, not committed)

KNOWN SECURITY FIXES (main):
  KB-001: Prevent role escalation via user sync push
  KB-002: Harden payment deep link against spoofing
  KB-005: Prevent 'INVnull' display for legacy bills
  KB-006: Audit trail for manual refund actions
  KB-007: Prevent UPI draft bill accumulation across sessions
  KB-008: Disable Swagger/OpenAPI in production
